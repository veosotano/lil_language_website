Design philosophy

This page describes the rules by which the design of the LIL language is guided. Keep in mind that the rules should be intepreted by its intentions (the spirit of the law) and not by the exact wording. They are just guidelines and, if necessary, can be broken.

If you have suggestions for something that we should add here, do not hesistate to get in touch.

When designing the syntax of LIL, keep in mind the following:

  1. Only use abbreviations if it is immediately clear what they mean, i.e. don't use cryptic ones.
  2. Whitespace should never be used as syntax.
  3. Be copy & paste friendly, e.g. allow extra separators at the end of a list.
  4. Keep names as short as possible, without being cryptic. If possible, use only one word.
  5. Use camelCase when joining two words in an identifier, use UPPER_CASE for built in operators.
  6. Do NOT worry about wether braces are on the same line or on a new one.
